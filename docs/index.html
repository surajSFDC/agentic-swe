// how it works

Three tracks, one pipeline

After feasibility, lean-track-check sets pipeline.track in state.json: lean (minimal), standard (design + tests, skips design panel and separate code-review phase), or rigorous (full panel, design-review, code-review, permissions). The Hypervisor (primary session) owns transitions and gates per CLAUDE.md.

Lean track — simple tasks
feasibility → lean-track-check → lean-track-implementation → validation → pr-creation → approval-wait → completed
Standard track — medium tasks
feasibility → lean-track-check → design → verification → test-strategy → implementation → self-review → validation → pr-creation → approval-wait → completed
Rigorous track — complex tasks
feasibility → lean-track-check → design → design-review → verification → test-strategy → implementation → self-review → code-review → permissions-check → validation → pr-creation → approval-wait → completed

■ Human gates stop the pipeline for your review. Nothing ships without approval.
■ Transition edges live in .claude/state-machine.json and match the fenced graph in CLAUDE.md (verified in CI).

// capabilities

What makes it different

⚙

Zero runtime code

The pipeline is markdown under .claude/ plus root CLAUDE.md. The host (e.g. Claude Code) executes it. Optional local tools like the brainstorm visual server are opt-in — not required to run /work.

◆

Evidence-based decisions

Every phase output follows a four-point evidence standard: observed, inferred, evidence, uncertain. No guessing.

★

135+ specialized agents

Auto-selected based on your codebase. Python, TypeScript, Rust, Kubernetes, security, ML — agents that know the domain.

☍

Human gates built in

The pipeline stops at ambiguity, approval, and escalation points. You stay in control of what ships.

⚙

Multi-agent review

Complex designs get a three-agent panel review: architect, security, and adversarial — running in parallel.

⚡

Track-aware routing

Lean, standard, and rigorous tracks share one state machine; lean-track-check records pipeline.track so phases and budgets match risk.

◉

Hypervisor session

The primary chat session is the Hypervisor: it carries state.json, runs mandatory /check steps, delegates to core agents and subagents, and stays accountable — not a separate hosted service.

// quick start

Up and running in 3 steps

1

Install into your project

npx agentic-swe /path/to/your/project

Or: npm install -g agentic-swe then agentic-swe install · Node 18+

2

Open Claude Code

cd /path/to/your/project && claude
3

Start working

/work Add retry logic to the API client
// commands

Key commands

/work <task>Start a new task — auto-routes lean, standard, or rigorous track
/work <id>Resume paused work by ID
/plan-only <task>Feasibility and design without implementing
/brainstormDesign-first exploration; optional visual server
/write-plan [id]Refine implementation plan — no coding
/execute-plan [id]Run the plan via implementation paths
/author-pipelineSafe checklist to extend phases and commands
/evaluate-work <id>Work item health and artifact status
/subagentBrowse, search, and invoke specialists
/repo-scanStructured codebase snapshot
/test-runner [scope]Detect and run tests
/lint [scope]Linters in check mode
/check budgetBefore phases — budgets intact
/check transitionBefore state changes — edge allowed for track
/check artifactsBefore transition — required files present
